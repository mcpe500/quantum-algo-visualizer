# SPEC 027: DJ Dynamic Explanation — Truth Table to Oracle

## Prompt

User meminta penjelasan yang jauh lebih jelas tentang bagaimana `truth_table` diubah menjadi oracle Deutsch-Jozsa, dengan constraint paling penting:

1. Penjelasan harus mengikuti dataset aktif.
2. Jika dataset baru ditambahkan, penjelasan harus otomatis ikut berubah.
3. Jika isi dataset diubah, penjelasan juga harus ikut berubah.
4. Penjelasan tidak boleh hard-coded per case seperti `DJ-01` atau `DJ-04`.

## Tujuan

1. Menutup gap pemahaman paling penting pada tab quantum DJ: hubungan antara dataset JSON dan oracle nyata.
2. Menjelaskan transformasi `truth_table -> blok oracle` secara manusiawi tetapi tetap teknis benar.
3. Menjaga implementasi tetap dinamis terhadap perubahan dataset tanpa kontrak API baru.

## Pemahaman Masalah

Sebelum perubahan ini, UI DJ sudah menampilkan:

1. `truth_table` penuh,
2. step animation,
3. highlight row aktif saat oracle berjalan.

Namun satu lapisan penting masih hilang: user bisa melihat **data** dan bisa melihat **oracle berjalan**, tetapi belum dijelaskan **aturan konversinya**. Akibatnya alur berpikir berikut belum eksplisit:

1. mana baris `f(x)=1` yang dipilih,
2. kapan `X` sementara dipakai,
3. kenapa `MCX` bisa mewakili satu pola input,
4. kenapa `X` harus dikembalikan lagi,
5. kenapa constant-zero dan constant-one tidak butuh blok oracle yang sama seperti balanced.

## Keputusan Teknis

### 1. Frontend-only derivation

Penjelasan oracle dibangun di frontend dari payload yang sudah ada:

1. `truth_table`
2. `oracle_summary`
3. `n_qubits`
4. `activeBits` dari langkah animasi aktif

Alasan:

1. Payload yang ada sudah cukup untuk menghitung penjelasan.
2. Tidak perlu ubah kontrak backend.
3. Risiko regresi paling rendah.
4. Dinamika dataset tetap terjaga penuh karena UI membaca data aktual, bukan teks statis.

### 2. Aturan derivasi dibuat sama dengan builder oracle backend

Logika penjelasan mengikuti `create_dj_circuit()` di backend:

1. `constant-zero`
   - tidak ada baris `f(x)=1`
   - oracle = identity
2. `constant-one`
   - semua baris `f(x)=1`
   - oracle = `X` pada ancilla sekali
3. `balanced`
   - hanya baris dengan `f(x)=1` yang diproses
   - setiap bit `0` pada input dibalik sementara dengan `X`
   - lalu jalankan `MCX(input -> ancilla)`
   - lalu kembalikan lagi `X` sementara

Dengan penyelarasan ini, penjelasan UI tidak sekadar edukatif, tetapi juga traceable ke implementasi circuit aktual.

### 3. Tambah panel khusus penjelasan

UI baru ditambahkan sebagai panel detail di sisi kanan animasi:

1. judul konsep,
2. intro ringkas yang menjelaskan profil dataset,
3. langkah algoritmik bernomor,
4. daftar blok oracle hasil derivasi per pola input bernilai `1`,
5. highlight dinamis untuk `activeBits` saat animasi sedang memproses pola tertentu.

## Logika Implementasi

### Helper baru: `buildOracleConstructionModel(data)`

Output model:

1. `intro`
2. `steps`
3. `rows`
4. `gateSummary`

#### Mode `constant-zero`

Hasil:

1. intro menjelaskan bahwa tidak ada input bernilai `1`
2. langkah menjelaskan oracle menjadi identity
3. `rows = []`

#### Mode `constant-one`

Hasil:

1. intro menjelaskan semua input bernilai `1`
2. langkah menjelaskan satu `X` pada ancilla cukup
3. `rows = []`

#### Mode `balanced`

Untuk setiap entry `truth_table` dengan `output === 1`:

1. ambil `inputBits`
2. cari semua posisi bit `0`
3. ubah posisi itu menjadi daftar label `q0`, `q1`, dst.
4. bentuk row model:
   - `zeroControlLabels`
   - `temporaryFlipLabels`
   - `controlLabels`
   - `mcxTargetLabel = ancilla`

### Pseudocode

```text
buildOracleConstructionModel(data):
  if profile == constant-zero:
    return identity explanation

  if profile == constant-one:
    return ancilla-X explanation

  rows = []
  for each truth_table entry:
    if output != 1:
      continue
    zeroControls = all qubit indices where input bit == 0
    rows.push({
      inputBits,
      zeroControlLabels,
      temporaryFlipLabels,
      controlLabels = all q labels,
      mcxTargetLabel = ancilla,
    })

  return balanced explanation with rows
```

## Kenapa Ini Dinamis

Implementasi ini dinamis karena:

1. tidak memeriksa `case_id` tertentu,
2. tidak memakai mapping manual untuk `DJ-01` sampai `DJ-04`,
3. jumlah blok oracle diambil langsung dari banyaknya baris `f(x)=1`,
4. label `zero-control` dihitung dari bit aktual setiap input,
5. jumlah kontrol mengikuti `data.n_qubits`,
6. highlight mengikuti `activeBits` dari langkah animasi backend.

Artinya:

1. tambah dataset baru -> panel ikut menyesuaikan,
2. ubah isi truth table -> row oracle berubah otomatis,
3. ubah jumlah qubit -> label kontrol dan daftar MCX ikut berubah.
