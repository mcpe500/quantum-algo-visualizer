# SPEC 020: DJ Video Quantum Export

## Prompt

User membutuhkan fitur `export as video` untuk animasi Deutsch-Jozsa. Scope yang dipilih adalah canvas animasi Three.js, bukan seluruh tab quantum. User juga menekankan bahwa hasilnya harus terasa seperti `video quantum`, sehingga video perlu tetap menjelaskan apa yang sedang terjadi agar dapat dipahami oleh orang yang belum akrab dengan komputasi kuantum.

Tambahan preferensi dari user:

1. format awal `WebM`,
2. `MP4` boleh dipikirkan sebagai langkah lanjutan,
3. implementasi diminta langsung dimulai.

## Tujuan

Menambahkan ekspor video pada animasi DJ dengan karakteristik berikut:

1. merekam scene Three.js yang sedang aktif,
2. menambahkan caption penjelasan ke dalam video,
3. menjaga hasil tetap sinkron dengan timeline nyata dari dataset,
4. tidak menambah pipeline berat yang merusak arsitektur frontend sekarang.

## Kenapa Solusi Ini Dipilih

Arsitektur DJ saat ini memisahkan dua lapisan dengan jelas. Scene utama dirender di canvas Three.js, sedangkan detail penjelasan, truth table, amplitudo, dan measurement diletakkan pada panel HTML di luar canvas. Karena itu, jalur paling aman untuk `export as video` bukan merekam seluruh DOM, melainkan merekam canvas utama lalu menambahkan penjelasan sebagai overlay pada canvas komposit.

Pendekatan ini dipilih karena tiga alasan. Pertama, inti visual yang benar-benar penting memang ada di scene Three.js: wire, kolom gate, orb, phase band, dan result board. Kedua, perekaman DOM penuh dengan `html2canvas` untuk banyak frame akan jauh lebih berat dan lebih rapuh, terutama karena ada kombinasi WebGL dan panel HTML. Ketiga, browser modern sudah menyediakan `MediaRecorder` dan `captureStream()` untuk canvas, sehingga fitur ini bisa dikirim tanpa menambah backend atau worker render baru.

## Keputusan Scope

### Implemented

1. `WebM` export dari canvas Three.js.

## Perubahan Logika

### 1. Export dilakukan dari canvas komposit

Alih-alih merekam DOM penuh, fitur baru membuat `compositorCanvas` tersembunyi. Setiap frame export:

1. canvas Three.js digambar ke kompositor,
2. overlay penjelasan digambar di atasnya,
3. kompositor direkam melalui `MediaRecorder`.

Dengan cara ini, output tetap berupa video yang menjelaskan proses quantum, tetapi tanpa membawa kompleksitas penuh dari perekaman seluruh halaman.

### 2. Overlay penjelasan dibagi tiga mode

Video dibagi ke tiga mode naratif:

1. `intro`
2. `play`
3. `outro`

`intro` menjelaskan aturan dasar pembacaan hasil Deutsch-Jozsa. `play` menjelaskan langkah aktif dari timeline aktual. `outro` menegaskan hasil measurement dan interpretasi akhir CONSTANT/BALANCED.

### 3. Export memakai playback terkendali

Sebelum export dimulai:

1. playback normal dihentikan,
2. step di-reset ke awal,
3. speed export dipaksa minimal cukup lambat untuk dibaca,
4. export merekam intro, playback utama, lalu outro.

Sesudah selesai:

1. speed lama dikembalikan,
2. posisi step lama dikembalikan,
3. status play lama dikembalikan bila sebelumnya sedang aktif.

### 4. Halaman dikunci saat export berjalan

Karena export terjadi secara asinkron dan bergantung pada state scene, kontrol luar seperti:

1. pemilihan case,
2. tab classic/quantum,
3. tombol run,
4. navigation link,

sementara dinonaktifkan saat export berjalan. Ini mencegah state berubah di tengah perekaman.

## Pseudocode Implementasi

```text
user click export video
  stop playback lama
  simpan state lama
  reset currentStep ke 0
  buat compositor canvas
  buat MediaRecorder dari compositor.captureStream()

  draw intro overlay beberapa saat

  set mode = play
  jalankan playback normal
  selama playback:
    draw threejs canvas ke compositor
    draw caption phase + operation + description

  force ke step terakhir
  set mode = outro
  draw hasil akhir beberapa saat

  stop recorder
  download blob .webm
  restore state lama
```

## Kenapa MP4 Ditunda

User sempat membuka kemungkinan `webm -> mp4 converter`. Secara teknis ini memungkinkan, tetapi fase itu sengaja belum diaktifkan pada sesi ini karena:

1. `ffmpeg.wasm` menambah beban bundle dan RAM secara signifikan,
2. conversion client-side dapat lambat pada perangkat biasa,
3. jalur WebM sudah cukup untuk menutup kebutuhan export video awal dengan risiko rendah.

Dengan kata lain, implementasi sekarang sengaja mengutamakan produk yang benar-benar jalan dan bisa diverifikasi, bukan fitur tambahan yang berat tetapi belum stabil.
