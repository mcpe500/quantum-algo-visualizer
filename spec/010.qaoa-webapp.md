# SPEC 010: QAOA Webapp Implementation

## Prompt

Bangun halaman QAOA yang menggantikan placeholder `/qaoa`, membaca dataset `datasets/qaoa/*.json`, dan menampilkan benchmarking Quantum Approximate Optimization Algorithm (QAOA) versus Simulated Annealing (SA) untuk Max-Cut. Implementasi harus memakai 1024 shots, memperlihatkan graph-to-Hamiltonian mapping, dan menampilkan optimizer trace serta circuit depth secara jelas.

## Purpose

Menjadikan QAOA sebagai studi kasus optimisasi kombinatorial yang mudah dibandingkan dengan pendekatan heuristik klasik. Visualisasi harus membantu membaca hubungan antara graph, Hamiltonian Ising, cost layer, mixer layer, hasil cut, dan kualitas solusi terhadap optimum eksak.

## Why

1. Proposal menempatkan QAOA sebagai representasi algoritma optimisasi aproksimasi.
2. Pembanding klasiknya adalah Simulated Annealing, sehingga kualitas solusi harus terlihat langsung.
3. Route `/qaoa` masih placeholder dan belum memakai dataset JSON yang sudah tersedia.
4. Dataset sudah memuat `nodes`, `edges`, dan `p_layers`, jadi backend dapat dibangun tanpa hardcoding.
5. QAOA perlu menampilkan optimizer trace, karena kualitas solusi ditentukan oleh iterasi parameter `gamma` dan `beta`.

## Changes Planned

### 1. Case discovery tetap data-driven

Backend membaca seluruh file `QAOA-*.json` dari folder dataset dan mengirimkannya ke frontend. Tidak ada daftar graph yang ditulis manual di kode.

### 2. Pembangunan problem Hamiltonian

Backend membaca `problem`, `graph.nodes`, `graph.edges`, dan `p_layers`. Graph harus dipetakan ke Hamiltonian Max-Cut/Ising, lalu digunakan untuk membangun cost layer dan mixer layer.

### 3. Benchmark QAOA versus SA

Sisi klasik menjalankan Simulated Annealing dan, untuk graf kecil, menghitung solusi eksak Max-Cut sebagai referensi kualitas. Sisi kuantum menjalankan QAOA dengan parameter `p_layers` dari dataset, lalu melaporkan cut terbaik, expected cut, depth, gate count, dan akurasi terhadap optimum.

### 4. Optimizer dan coupling map

Optimizer klasik harus konsisten antar framework. Referensi implementasi dapat memakai COBYLA dengan batas iterasi dan toleransi yang sama. Transpiled circuit harus menunjukkan pengaruh coupling map, termasuk kemungkinan SWAP yang menambah depth.

### 5. Frontend QAOA page

Halaman QAOA mengikuti pola DJ: case selector, tab klasik/kuantum, panel metrik, visualisasi graph, grafik optimizer trace, visualisasi sirkuit, dan section perbandingan. Tab klasik menampilkan graph Max-Cut dan hasil SA, sedangkan tab kuantum menampilkan Hamiltonian Ising, ansatz QAOA, dan kualitas cut.

### 6. Endpoint yang dibutuhkan

- `GET /api/qaoa/cases`
- `POST /api/qaoa/benchmark`
- `GET /api/qaoa/circuit-image/<case_id>`
- `GET /api/qaoa/trace/<case_id>`
- `GET /api/qaoa/dataset/<case_id>`

## Logic / Pseudocode

```text
scan datasets/qaoa/*.json
load selected case
extract nodes, edges, and p_layers
build graph and Max-Cut Hamiltonian

classical side:
  run simulated annealing on the same graph
  compute exact cut for small graphs

quantum side:
  build QAOA circuit from cost layer and mixer layer
  bind gamma and beta parameters
  transpile per framework and coupling map
  optimize parameters with 1024 shots per evaluation

benchmark result:
  report best cut, expected cut, exact cut, depth, gate count, and accuracy
  return per-framework summary
```
