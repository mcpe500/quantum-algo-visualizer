# SPEC 063: DJ Bloch Arrow Gate Occlusion

## Prompt

User reported issue from `datasets/dj/issue/issue.md`:

- Bloch sphere arrow direction is still unclear.
- Arrow becomes visible only in a better angle/position.
- When the state is at a gate, the arrow becomes hard to see.

## Root Cause

The Bloch sphere was placed at the same `x` and `y` position as the active gate. In DJ animation, the active gate is rendered in the same column and lane as the Bloch node. This caused the gate box, sphere mesh, ring, and arrow to visually overlap. Increasing arrow size alone was not enough because the active gate could still sit directly inside the sphere footprint.

## Implementation

1. **DJ-only state position offset**
   - Added a forward `sphereOffset` in `frontend/src/components/dj/animation/scene-primitives.tsx`.
   - The Bloch sphere now appears slightly after the active gate column.
   - Offset is bounded so final measurement does not collide with the result board.

2. **Arrow visibility hardening**
   - Updated `frontend/src/shared/components/3d/gates/BlochQubitNode.tsx`.
   - Arrow uses larger head/shaft.
   - Arrow uses dark `meshBasicMaterial`.
   - Arrow uses `depthTest={false}` and `depthWrite={false}` so it remains readable over sphere/gate overlap.
   - Sphere is more transparent and does not write depth.
   - Equator ring is muted so it no longer competes with the actual arrow.

## Why This Fix

The DJ-only offset fixes the real overlap between gate and Bloch sphere. The shared arrow hardening fixes the remaining readability problem when camera angle, sphere wireframe, or nearby scene objects reduce contrast.

## Acceptance Criteria

- Active gate remains visible.
- Bloch sphere no longer sits directly on top of the gate center.
- Arrow direction remains visible in H, X, MCX control/target, and measurement steps.
- Existing dataset/statevector logic remains unchanged.
