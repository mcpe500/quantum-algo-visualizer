# SPEC 064: Frontend + Backend JSON-Driven Refactor

## Prompt

User meminta refactor seluruh codebase yang relevan di:

1. `frontend/`
2. `backend/`

Dengan constraint keras:

1. Ikuti `spec/prompts/INSTRUCTIONS.md`.
2. Ikuti `spec/prompts/BEHAVIOUR.md`.
3. Gunakan ultrathink dan sequential-thinking.
4. Jangan cheating.
5. Definisi cheating dari user: hardcode data padahal sudah ada dataset JSON. Implementasi harus mengikuti JSON, bukan membuat nilai kasus, graph, signal, truth table, molecule, atau case list manual.
6. Buat spec file dulu saja untuk direview sebelum implementasi.

## Tujuan

Membuat refactor plan agar frontend dan backend menjadi lebih data-driven, maintainable, dan konsisten dengan dataset JSON sebagai sumber kebenaran utama.

Tujuan spesifik:

1. Menghapus hardcoded case defaults seperti `DJ-01`, `QFT-01`, `VQE-01`, dan `QAOA-01` dari flow runtime jika data tersebut bisa diambil dari JSON lewat `/cases` atau `list_cases()`.
2. Menghapus template data algoritma yang ditulis manual di frontend ketika data yang sama sudah tersedia di dataset JSON.
3. Mengubah script backend yang saat ini menyebut case list manual menjadi membaca case dari folder `datasets/<algorithm>/*.json`.
4. Menjaga seluruh behavior algoritma tetap berasal dari data JSON: DJ dari truth table, QFT dari signal data, VQE dari raw/canonical molecule data, QAOA dari adjacency matrix.
5. Memastikan lint/build/backend syntax/timeline checks tetap berjalan tanpa men-disable validasi.

## Kenapa Perlu

Project ini adalah visualizer algoritma quantum berbasis dataset. Jika UI/backend memakai hardcoded case atau graph, maka visualisasi bisa terlihat benar untuk dataset saat ini tetapi gagal ketika JSON berubah, dataset baru ditambah, atau data lama diperbaiki.

Dampak hardcode yang harus dihindari:

1. Dataset baru tidak otomatis muncul sebagai default.
2. UI bisa menampilkan graph atau nilai yang tidak sama dengan JSON.
3. Backend scripts bisa melewatkan case baru karena case list ditulis manual.
4. Review akademik bisa menganggap visualizer tidak jujur terhadap dataset.
5. Test bisa pass untuk data lama tetapi bukan karena sistem benar-benar membaca JSON.

## Validated Findings

### 1. Frontend hooks still hardcode initial cases

Files:

1. `frontend/src/hooks/useDJBenchmark.ts`
2. `frontend/src/hooks/useQFT.ts`
3. `frontend/src/hooks/useVQE.ts`
4. `frontend/src/hooks/useQAOA.ts`

Current pattern:

```ts
const [selectedCaseId, setSelectedCaseId] = useState<string>('QFT-01');
const [availableCases, setAvailableCases] = useState<string[]>(['QFT-01']);
```

Why it is a problem:

1. UI starts from a manual case ID before `/cases` returns real JSON-backed cases.
2. Effects can call circuit/trace/animation endpoints using the manual default.
3. If first JSON case changes or only another case exists, the app can make invalid calls before correcting state.

Target behavior:

1. Initial `selectedCaseId` should be empty or unresolved.
2. Cases must load from API first.
3. First selected case must be `sortCaseIds(ids)[0]` from API result.
4. API calls must be guarded until `selectedCaseId` exists.

### 2. Shared hook factory still encodes hardcoded default case pattern

File:

1. `frontend/src/shared/hooks/useAlgorithmBenchmark.ts`

Current pattern:

```ts
interface AlgorithmBenchmarkConfig {
  defaultCaseId: string;
}
```

Why it is a problem:

1. It makes future hooks repeat the same hardcoded default-case anti-pattern.
2. It conflicts with dataset JSON as source of truth.

Target behavior:

1. Remove required `defaultCaseId` from config, or make it optional only as a last-resort UI fallback.
2. Prefer `api.getCases()` as the source for initial selection.
3. Guard all `runBenchmark`, `loadCircuitImage`, `loadTrace`, `loadAnimation`, and `download` operations when no case is selected.

### 3. QAOA Simulated Annealing flow contains hardcoded JSON templates

File:

1. `frontend/src/components/qaoa/QAOASimulatedAnnealingFlow.tsx`

Current pattern:

```ts
const templates = {
  K3: `{
    "case_id": "QAOA-01",
    "graph": { "adjacency_matrix": [[0,1,1], ...] }
  }`,
  K4: `{
    "case_id": "QAOA-02",
    "graph": { "adjacency_matrix": [[0,1,1,1], ...] }
  }`,
};
```

Why it is a problem:

1. The same graph data exists in `datasets/qaoa/QAOA-01.json` and `datasets/qaoa/QAOA-02.json`.
2. If dataset JSON changes, UI templates become stale.
3. This is exactly the user-defined cheating category: hardcoded dataset values instead of following JSON.

Target behavior:

1. Remove hardcoded K3/K4 templates.
2. Use `createTemplateFromResult(result)` for current benchmark result.
3. If example switching is still required, load examples from `qaoaApi.getCases()` or pass dataset-backed cases from the page/hook.
4. JSON textarea may still support manual paste, but built-in examples must come from dataset/API.

### 4. Backend routes hardcode default case IDs

Files:

1. `backend/api/routes/dj.py`
2. `backend/api/routes/qft.py`
3. `backend/api/routes/vqe.py`
4. `backend/api/routes/qaoa.py`

Current pattern examples:

```py
case_id = data.get('case_id', 'DJ-01')
case_id = data.get('case_id', 'QFT-01')
case_id = data.get('case_id', 'VQE-01')
case_id = data.get('case_id', 'QAOA-01')
```

Why it is a problem:

1. Backend behavior depends on manual IDs instead of available JSON files.
2. If `DJ-01.json` is removed or renamed but another DJ JSON exists, omitted case requests break incorrectly.
3. Runtime source of truth should be `get_*_cases()` from dataset listing.

Target behavior:

1. Add a shared helper to resolve omitted case IDs from real dataset-backed case lists.
2. If request includes `case_id`, use it unchanged and validate through service load.
3. If request omits `case_id`, use the first sorted case from `get_*_cases()`.
4. If no dataset exists, return a clear JSON error and 404 or 400 depending endpoint semantics.

### 5. Backend scripts hardcode case lists and local paths

Files:

1. `backend/generate_qft_images.py`
2. `backend/generate_vqe_qaoa_images.py`
3. `backend/test_classic_run.py`
4. `backend/test_classic.py`
5. `backend/debug_dj.py`

Current problems:

1. `generate_qft_images.py` loops over `['QFT-01', 'QFT-02']`.
2. `generate_vqe_qaoa_images.py` hardcodes VQE circuit dimensions and QAOA K3 edges instead of reading VQE/QAOA JSON/service outputs.
3. `test_classic_run.py` loops over `['DJ-01', 'DJ-02', 'DJ-03', 'DJ-04']` and uses absolute dataset path.
4. Some scripts use local `D:\Ivan\...` paths.

Why it is a problem:

1. New JSON files are ignored.
2. Scripts are not portable.
3. Generated images can be detached from the actual dataset.

Target behavior:

1. Scripts should discover cases using `services.common.list_cases()` or service-level `get_*_cases()`.
2. Scripts should derive graph edges, qubit counts, ansatz layers, and signal sizes from JSON/service outputs.
3. Output directory should be CLI argument or relative default, not absolute local path.
4. Ad-hoc tests should become read-only checks or clearly named scripts if they write generated artifacts.

### 6. QFT service contains unreachable timeline code

File:

1. `backend/services/qft_service.py`

Current issue:

1. `build_qft_animation_timeline()` returns at line around 358.
2. A second large timeline-building block exists after that return.
3. The unreachable block references `Operator` without import and can mislead future changes.

Why it matters for this task:

1. Dead code makes it unclear which timeline follows JSON-derived signal data.
2. Refactor should preserve the live JSON-driven path and remove unreachable alternatives.

Target behavior:

1. Keep only the live timeline path that derives amplitudes from `signal_data`.
2. Remove unreachable block after verifying `validate_qft_timeline.py` still passes.

## Implementation Plan

### Phase 1: Shared dataset/default case helpers

Backend new helper candidate:

```py
def first_case_id(cases):
    if not cases:
        return None
    return cases[0].get('case_id')

def resolve_case_id(requested_case_id, cases):
    if requested_case_id:
        return requested_case_id
    return first_case_id(cases)
```

Frontend helper candidate:

```ts
function caseIdsFromCases(cases: Array<{ case_id?: string }>) {
  return sortCaseIds(cases.map((item) => item.case_id).filter(Boolean));
}

function firstDatasetCaseId(cases) {
  return caseIdsFromCases(cases)[0] ?? '';
}
```

Expected result:

1. No hook initializes with algorithm-specific hardcoded `*-01` case.
2. No route defaults to algorithm-specific hardcoded `*-01` case.
3. When case list is empty, UI shows empty/loading state and backend returns clear error.

### Phase 2: Frontend hooks become JSON-first

Update files:

1. `frontend/src/hooks/useDJBenchmark.ts`
2. `frontend/src/hooks/useQFT.ts`
3. `frontend/src/hooks/useVQE.ts`
4. `frontend/src/hooks/useQAOA.ts`
5. `frontend/src/shared/hooks/useAlgorithmBenchmark.ts`

Logic:

```ts
selectedCaseId = ''
availableCases = []

on mount:
  cases = await api.getCases()
  ids = sort case ids from API response
  availableCases = ids
  selectedCaseId = first id if current missing

on selectedCaseId change:
  if no selectedCaseId: skip API calls
  load circuit/trace/animation from selectedCaseId

on run:
  if no selectedCaseId: show error
  else run benchmark with selectedCaseId
```

Why:

1. The first selected case now follows JSON-backed `/cases` response.
2. No early call to hardcoded `*-01` occurs before dataset load.

### Phase 3: QAOA Simulated Annealing templates become dataset-backed

Update file:

1. `frontend/src/components/qaoa/QAOASimulatedAnnealingFlow.tsx`

Option A, minimal and safest:

1. Remove `templates.K3` and `templates.K4`.
2. Keep only current result template using `createTemplateFromResult(result)`.
3. Rename UI copy from “Pilih contoh graph” to “Gunakan JSON dari benchmark saat ini atau paste JSON lain”.

Option B, richer but still data-driven:

1. Load all QAOA cases through `qaoaApi.getCases()`.
2. Build example buttons/dropdown from API cases.
3. When user chooses one case, populate textarea from that case JSON.

Recommended choice:

Option A for first implementation because it removes cheating immediately with minimal risk. Option B can follow if user wants multi-case examples inside the SA panel.

### Phase 4: Backend routes resolve omitted case from JSON

Update files:

1. `backend/api/routes/dj.py`
2. `backend/api/routes/qft.py`
3. `backend/api/routes/vqe.py`
4. `backend/api/routes/qaoa.py`

Logic:

```py
data = request.get_json(silent=True) or {}
case_id = resolve_case_id(data.get('case_id'), get_algorithm_cases())
if case_id is None:
    return jsonify({'error': 'No dataset cases available'}), 404
payload = service(case_id)
```

Why:

1. Backend default follows dataset folder contents.
2. Existing explicit `case_id` requests still behave the same.

### Phase 5: Backend scripts become JSON-discovered

Update files:

1. `backend/generate_qft_images.py`
2. `backend/generate_vqe_qaoa_images.py`
3. `backend/test_classic_run.py`
4. `backend/test_classic.py`
5. `backend/debug_dj.py`

Logic:

```py
cases = get_qft_cases()
for case in cases:
    case_id = case['case_id']
    derive n_qubits from signal_data/n_points
    generate output
```

For QAOA:

```py
cases = get_qaoa_cases()
for case in cases:
    matrix = case['graph']['adjacency_matrix']
    edges = matrix_to_edges(matrix)
    build circuit from edges
```

For VQE:

```py
cases = get_vqe_cases()
for case in cases:
    n_qubits = case['preprocessing']['target_qubits'] or canonical n_qubits
    ansatz_type = case['experiment']['ansatz_type']
    n_layers = case['experiment']['n_layers']
    build circuit from dataset values
```

Why:

1. Scripts no longer assume exactly two QFT cases, two VQE cases, or K3 QAOA.
2. Output is traceable to JSON.

### Phase 6: QFT unreachable cleanup

Update file:

1. `backend/services/qft_service.py`

Change:

1. Delete unreachable block after first `return timeline`.
2. Keep active implementation that builds timeline from `signal_data`.

Why:

1. Reduces ambiguity about which code path is authoritative.
2. Prevents future accidental revival of stale hardcoded/duplicated logic.

## Pseudocode: End-to-End Data Flow After Refactor

### Frontend load flow

```text
Page mounts
Hook calls /api/<algorithm>/cases
Backend lists datasets/<algorithm>/*.json
Backend returns sorted case objects
Frontend extracts case_id list
Frontend selects first returned case id
Frontend loads trace/circuit/animation using selected case id
UI renders only data from API payload
```

### Backend omitted-case flow

```text
POST /api/qft/benchmark with no case_id
Route calls get_qft_cases()
get_qft_cases() reads datasets/qft/QFT-*.json
Route picks first sorted case_id from returned JSON cases
run_qft_payload(case_id, shots) loads that JSON
Response is derived from signal_data in selected JSON
```

### QAOA SA flow after removing hardcoded templates

```text
QAOA benchmark result arrives
SA panel builds default textarea JSON from result.case_id, result.problem, result.description, result.adjacency_matrix
User runs simulation
Parser reads textarea JSON
Simulation uses adjacency_matrix from JSON only
No built-in K3/K4 literal exists in component
```

## Edge Case Analysis

### Empty dataset folder

Expected behavior:

1. Frontend `availableCases = []`.
2. Run button should not call benchmark without selected case.
3. Backend omitted-case request returns clear error.

### Dataset case ID does not start at `*-01`

Expected behavior:

1. First sorted JSON case becomes default.
2. No `*-01` request is made unless user explicitly selects it and it exists.

### Dataset JSON changes graph/signal/truth table

Expected behavior:

1. UI and backend reflect updated JSON on reload.
2. No stale frontend template overrides the JSON.

### User pastes custom QAOA JSON in SA panel

Expected behavior:

1. Manual paste remains allowed because it is explicit user input, not hidden hardcode.
2. Built-in examples must come from current result or API dataset.

### Invalid shots/query parameters

Expected behavior:

1. Parser returns default or clear 400 depending endpoint decision.
2. No uncaught `ValueError` from raw `int()` or `float()` conversion.

## No-Cheating Rules For Implementation

1. Do not replace hardcoded data with another hidden hardcoded data source.
2. Do not keep hardcoded JSON templates that duplicate files under `datasets/`.
3. Do not fake API responses in production runtime.
4. Do not skip `/cases` by assuming first case ID.
5. Do not disable ESLint, TypeScript, Playwright, or Python checks to pass.
6. Do not delete tests or routes to avoid fixing failures.
7. Do not ignore failing verification. If a command fails, record and fix root cause.
