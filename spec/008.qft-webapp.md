# SPEC 008: QFT Webapp Implementation

## Prompt

Bangun halaman QFT yang menggantikan placeholder `/qft`, membaca dataset `datasets/qft/*.json`, dan menampilkan perbandingan Quantum Fourier Transform (QFT) versus Fast Fourier Transform (FFT) secara data-driven. Implementasi harus tetap memakai 1024 shots, mendukung Qiskit, Cirq, dan ProjectQ, serta memperlihatkan dampak padding dan coupling map terhadap circuit depth.

## Purpose

Menjadikan QFT sebagai studi kasus transformasi domain frekuensi yang dapat divalidasi secara visual dan kuantitatif. Halaman ini harus memperlihatkan hubungan antara sinyal diskret, spektrum FFT, sirkuit QFT, dan hasil benchmark antar framework, sehingga pembaca dapat melihat bagaimana struktur sirkuit memengaruhi kompleksitas dan akurasi hasil.

## Why

1. Proposal menempatkan QFT sebagai wakil kategori aljabar dan teori bilangan.
2. Pembanding klasiknya adalah FFT, sehingga QFT perlu divisualisasikan sebagai pasangan analisis yang setara.
3. Route `/qft` masih placeholder dan belum membaca dataset JSON yang sudah tersedia.
4. Dataset QFT bersifat signal-based, sehingga backend harus menghitung hasil dari `signal_data` tanpa hardcoding.
5. Perubahan ukuran input harus tercermin langsung pada jumlah qubit, gate count, dan circuit depth.

## Changes Planned

### 1. Case discovery tetap data-driven

Backend membaca seluruh file `QFT-*.json` dari folder dataset dan mengekspose daftar case ke frontend. Tidak ada daftar case yang ditulis manual di kode.

### 2. Normalisasi input sinyal

Backend memakai `n_points`, `signal_data`, dan `signal_type` dari JSON. Jika panjang sinyal bukan pangkat dua, backend melakukan padding ke pangkat dua terdekat di atasnya sebelum membangun statevector dan sirkuit QFT.

### 3. Benchmark QFT versus FFT

Sisi klasik menghitung FFT dari sinyal yang sama dan mengekstrak bin dominan serta magnitudo spektrum. Sisi kuantum membangun sirkuit QFT dengan gerbang Hadamard, controlled-phase, SWAP, dan measurement. Hasil benchmark harus memuat `time_complexity`, `circuit_depth`, `gate_count`, `shots`, dan `accuracy`.

### 4. Pengaruh coupling map dan SWAP

Transpiled circuit harus merefleksikan batasan konektivitas qubit. Jika framework membutuhkan rute tambahan, penambahan SWAP harus terlihat pada depth dan gate count akhir.

### 5. Frontend QFT page

Halaman QFT mengikuti pola DJ: case selector, tab klasik/kuantum, panel metrik, visualisasi sirkuit, tabel tahap sirkuit, dan section perbandingan. Tab klasik menampilkan sinyal dan spektrum FFT, sedangkan tab kuantum menampilkan sirkuit QFT, trace tahap, dan ringkasan hasil benchmark.

### 6. Endpoint yang dibutuhkan

- `GET /api/qft/cases`
- `POST /api/qft/benchmark`
- `GET /api/qft/circuit-image/<case_id>`
- `GET /api/qft/trace/<case_id>`
- `GET /api/qft/dataset/<case_id>`

## Logic / Pseudocode

```text
scan datasets/qft/*.json
load selected case
extract signal_data and n_points
if n_points is not power of two:
  pad to next power of two

classical side:
  compute FFT on the same signal
  identify dominant frequency bins

quantum side:
  build QFT circuit for the padded size
  apply H, controlled-phase, and SWAP layers
  transpile per framework and coupling map
  execute with 1024 shots

benchmark result:
  compare dominant bins / spectral shape
  record depth, gate count, and accuracy
  return per-framework summary
```
