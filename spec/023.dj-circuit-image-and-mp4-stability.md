# SPEC 023: DJ Circuit Image Fallback + MP4 Conversion Stability

## Prompt

User melaporkan dua masalah produksi:
1. Endpoint `dj/circuit-image` dan `dj/circuit-image-boxed` sering `500`.
2. Export MP4 stuck di status loading (spinner tidak selesai).

## Tujuan

1. Membuat endpoint gambar sirkuit DJ tetap menghasilkan output walau renderer MPL gagal.
2. Menghilangkan kondisi "loading tanpa akhir" pada konversi MP4.
3. Meningkatkan visibilitas error agar frontend menampilkan pesan backend asli.

## Akar Masalah

### A. Circuit image 500

`get_dj_circuit_image_payload()` sebelumnya hanya mengandalkan `circuit_drawer(..., output='mpl')`.
Jika dependency optional/renderer MPL gagal, endpoint langsung 500.

### B. MP4 endless loading

`videoConvert.ts` sebelumnya tidak punya timeout untuk:
- `ffmpeg.load(...)`
- `ffmpeg.exec(...)`

Jika CDN lambat/blocked atau transcoding hang, Promise tidak selesai dan UI terlihat loading terus.

## Keputusan Teknis

### 1. Backend fallback renderer

Di `dj_service.py`, jika MPL drawer gagal:
- fallback ke text-circuit rendering (`qc.draw(output='text')`) di-embed ke figure matplotlib monospace.
- endpoint tetap return `image` base64 (bukan error 500).

Tambahan metadata:
- `render_mode`: `mpl` atau `text-fallback`
- `warning` jika fallback aktif

Tambahan patch stabilisasi:
- Jika `circuit_drawer(..., output='mpl')` berhasil tetapi gagal saat `savefig`/`figure_to_base64`, sistem otomatis retry dengan text-fallback.
- Ini menutup kasus `ParseException` yang muncul di fase save, bukan di fase draw.

### 2. MP4 conversion timeout + CDN fallback

Di `videoConvert.ts`:
- Tambah helper `withTimeout(...)`
- Timeout load core: `45s`
- Timeout exec convert: `120s`
- Coba CDN berurutan: jsDelivr lalu unpkg
- Reset `ffmpegLoading` jika load gagal supaya retry berikutnya bisa jalan

Tambahan patch stabilisasi:
- Timeout juga diterapkan saat `toBlobURL(...)` (download `ffmpeg-core.js` dan `ffmpeg-core.wasm`), sehingga loading tidak bisa menggantung sebelum `ffmpeg.load(...)` dipanggil.

### 3. Error message propagation di frontend API

Di `frontend/src/services/api.ts`:
- Tambah `readApiError(response, fallback)`
- `getCircuitImage` dan `getCircuitImageBoxed` sekarang throw error backend asli bila tersedia

## File yang Diubah

| File | Perubahan |
|------|-----------|
| `backend/services/dj_service.py` | fallback text render untuk circuit image + metadata warning/render_mode |
| `frontend/src/utils/videoConvert.ts` | timeout load/exec, fallback CDN, convert command lebih kompatibel |
| `frontend/src/services/api.ts` | parse error backend agar message tidak generik |
| `frontend/src/components/dj/DJQuantumAnimation.tsx` | update wording validasi MP4 support |

## Pseudocode

```text
backend get_dj_circuit_image_payload(case_id, boxed):
  qc = create_dj_circuit(...)
  try mpl_drawer:
    fig = circuit_drawer(..., output='mpl')
    render_mode = 'mpl'
  except:
    fig = render_text_circuit(qc)
    render_mode = 'text-fallback'
    warning = reason

  if boxed and render_mode == 'mpl':
    try add_phase_boxes(fig, qc)
    except: warning = overlay reason

  return { image: base64(fig), render_mode, warning? }
```

## Risiko Tersisa

1. Jika jaringan benar-benar memblokir kedua CDN FFmpeg core, MP4 tetap gagal (sekarang fail-fast dengan timeout).
2. Fallback text renderer tidak seindah MPL drawer, tetapi menjaga endpoint tetap hidup.
